The Policy Node: The "Business Constitution" Spec

In SurrealDB, we don't want a flat text file. We want a Graph-Based SOP (Standard Operating Procedure) that an agent can traverse like a map.

1. The Schema (SurrealQL)

SQL
DEFINE TABLE policy SCHEMAFULL;

-- The Core Identification
DEFINE FIELD title ON policy TYPE string;
DEFINE FIELD version ON policy TYPE string; -- e.g. "2026.03.10"
DEFINE FIELD status ON policy TYPE string ASSERT $value IN ['active', 'draft', 'deprecated'];

-- The Logic: Pre-conditions and Constraints
DEFINE FIELD scope ON policy TYPE array; -- e.g. ["finance", "hiring", "api_access"]
DEFINE FIELD rules ON policy TYPE array; 
/* Example Rule Object:
{
  id: "max_spend_limit",
  condition: "intent.budget_limit.amount <= 500",
  effect: "permit",
  priority: 1
}
*/

-- The "Veto" Configuration
DEFINE FIELD requires_human_approval ON policy TYPE bool DEFAULT false;
DEFINE FIELD approval_timeout ON policy TYPE duration DEFAULT 10m;

-- Traceability
DEFINE FIELD created_at ON policy TYPE datetime DEFAULT time::now();
DEFINE FIELD updated_at ON policy TYPE datetime DEFAULT time::now();
2. The "Graph-SOP" Relationship

The power of a Graph is that Policies can be Nested. * A Global Finance Policy might point to a Marketing Budget Policy.

When the Authorizer Agent checks an Intent, it doesn't just look at one node; it traverses the path of policies to ensure no contradictions.

SQL
-- Link a specific Agent/Identity to a Policy
DEFINE TABLE governed_by TYPE RELATION IN identity OUT policy SCHEMAFULL;

-- Link a Policy to a specific Project/Workspace
DEFINE TABLE applied_to TYPE RELATION IN policy OUT workspace SCHEMAFULL;
3. How the "Authorizer Agent" uses the Policy Node

This is the "Process Layer" in action. When your Orchestrator triggers the Authorizer, the prompt changes from "Is this okay?" to:

The Auditor's Directive:

Fetch all policy nodes linked to identity:worker-agent.

Fetch all policy nodes linked to workspace:current-project.

Compare the intent.action_spec against every rule in these policies.

Hard Fail: If any rule with effect: "deny" is triggered, reject the intent immediately.

Soft Fail: If requires_human_approval is true, move status to pending_human_veto and alert the Governance Feed.

4. Why this beats the "Status Quo"

Feature	Prompt-Based Rules	Brain (Policy-as-Graph)
Enforcement	Suggestions (Can be ignored).	Architectural (Hard-coded blocks).

Paperclip and others are building "Prompt Libraries." You are building a Deterministic Governance Layer. If a Human CEO says, "Stop all spending on LinkedIn ads until Monday," you don't have to go through every agent's system prompt. In a graph-native Business OS, the Policy Node isn't a text document. It is a deterministic set of constraints that the Authorizer Agent uses to evaluate an Intent.

The Policy Node Specification (v1.0)

The goal of this node is to provide a machine-verifiable source of truth for what is allowed, who can do it, and under what conditions.

1. The Schema (SurrealQL)

We use a SCHEMAFULL table to ensure that no "loose" logic enters the governance layer.

SQL
DEFINE TABLE policy SCHEMAFULL;

-- Core Metadata
DEFINE FIELD title ON policy TYPE string;
DEFINE FIELD description ON policy TYPE string;
DEFINE FIELD version ON policy TYPE string; 
DEFINE FIELD status ON policy TYPE string 
    ASSERT $value IN ['active', 'draft', 'deprecated', 'testing'];

-- Targeted Scope (Where does this policy apply?)
DEFINE FIELD selector ON policy TYPE object; 
/* Example: { workspace: "finance", agent_role: "accountant", resource: "stripe" } */

-- The Rules Engine
-- We use an array of objects to allow multiple logic gates per policy
DEFINE FIELD rules ON policy TYPE array;
/* Rule Structure:
{
  id: string,
  effect: "allow" | "deny",
  condition: string, -- A SurrealQL snippet or logic string
  priority: int
}
*/

-- Authorization Requirements
DEFINE FIELD require_mfa ON policy TYPE bool DEFAULT false;
DEFINE FIELD human_veto_required ON policy TYPE bool DEFAULT false;
DEFINE FIELD max_ttl ON policy TYPE duration DEFAULT 1h; -- Max life for the RAR token

-- Auditing
DEFINE FIELD created_by ON policy TYPE record<identity>;
DEFINE FIELD updated_at ON policy TYPE datetime DEFAULT time::now();
2. The Relationship Graph

The Policy Node sits between an Identity and a Resource. In SurrealDB, we represent this as a direct path the Authorizer can traverse.

SQL
-- Link an Identity (Person/Agent) to a Policy
DEFINE TABLE governing TYPE RELATION IN identity OUT policy SCHEMAFULL;

-- Link a Policy to the Resources/Workspaces it protects
DEFINE TABLE protects TYPE RELATION IN policy OUT workspace SCHEMAFULL;
3. The "Process Layer" Workflow

Now that your RAR + DPoP is ready, the flow looks like this:

Request: An Agent generates an Intent (e.g., "Pay Invoice #123").

Lookup: The Authorizer Agent queries the graph:
SELECT ->governing->policy.* FROM identity:current_agent

Evaluation: The Authorizer checks the intent.action_spec against the policy.rules.

RAR Issuance:

If Pass: The Authorizer generates an OAuth 2.1 RAR (Rich Authorization Request) object containing the specific, narrowed scopes defined in the policy.

If Fail: The Intent is moved to status: "vetoed".

DPoP Binding: The resulting token is bound to the Agent's DPoP key, ensuring that even if the token is intercepted, it's useless without the Brain's private key.

4. Implementation Example: "The $500 Limit"

If you want to enforce a spend limit, you don't write it in a prompt. You write it in the node:

Policy Name: Finance-Small-Spend
Rule: IF intent.amount < 500 THEN effect: "allow"
Selector: { resource: "banking_api" }

Why this wins: If the agent tries to pay $501, the RAR token is never generated. The agent doesn't even get the chance to "talk" to the bank.

The "Brutally Honest" Verdict

By building this Policy Node, you have officially moved out of the "AI Chatbot" category and into the "Autonomous Governance" category.

Paperclip solves for "Can we do this?" (Capability).
Brain solves for "Are we allowed to do this?" (Authority).

**Phase 1 -- Jobs-to-be-Done Analysis (REQUIRED):**

Grounds all subsequent artifacts in real user motivations.

1. **Job Discovery**: Ask user what users are trying to accomplish. Capture in job story format: "When [situation], I want to [motivation], so I can [outcome]."
2. **Job Dimensions**: For each job — functional (practical task)|emotional (desired feeling)|social (desired perception)
3. **Four Forces Analysis**: For each primary job:
   - **Push** (current frustration): "What frustrated users enough to request this?"
   - **Pull** (desired future): "What could they do that they cannot now?"
   - **Anxiety** (adoption concerns): "What concerns about adopting this?"
   - **Habit** (current behavior): "What behavior must change?"
   If interview transcripts|support tickets|analytics exist, extract forces from those instead of relying solely on user description.
4. **Opportunity Scoring** (multiple jobs): Rank by importance vs. satisfaction gap. High importance + low satisfaction = strongest opportunities. Produce scored table.
5. **JTBD-to-Story Bridge**: Each job story feeds into user stories and acceptance criteria in Phase 3. Every user story must trace to at least one job.
