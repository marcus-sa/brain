Non-atomic two-step write can leave observations in a partial state

persistObservation first creates the observation record via createObservation, then issues a separate UPDATE to set verified and source. If the UPDATE call on line 504 fails (network blip, DB timeout, transaction conflict), the observation will exist in the database with verified = false (default) and source = null, permanently silently incorrect — the verification result will not match what is stored.

Since SurrealDB supports setting fields on INSERT/CREATE, both fields should be passed to createObservation directly (or the createObservation helper should accept them), making the write atomic and eliminating the separate UPDATE.
