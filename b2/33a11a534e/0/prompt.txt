Version monotonicity check is outside the transaction (TOCTOU)

The version guard on lines 157–166 reads oldVersion in a separate query before the transaction block begins at line 168. If two different draft policies (both with supersedes = X) are activated concurrently, both reads can return oldVersion = 1, both checks pass, and both transactions execute successfully. The result is two policies with status = 'active' in the same version chain, violating the single-active-version invariant that ADR-005 is designed to protect.

To close the window, the monotonicity assertion should be expressed as a conditional guard inside the transaction, so the check and the status updates execute atomically. SurrealDB supports LET variables and IF … THROW within a transaction block, which would allow the version comparison to be done without a separate round-trip:

BEGIN TRANSACTION;
  LET $oldVer = (SELECT VALUE version FROM $oldPolicy)[0];
  IF $oldVer >= $policyVersion {
    THROW string::concat("version ", <string> $policyVersion,
      " must be greater than superseded version ", <string> $oldVer);
  };
  UPDATE $policy SET status = 'active', updated_at = time::now();
  -- rest of the statements …
COMMIT TRANSACTION;
As written, activatePolicy is called from handleActivatePolicy which first fetches the policy record outside a lock, so a second concurrent caller can also enter this path for the same $oldPolicy before the first transaction commits.
