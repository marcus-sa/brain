Non-atomic version number: duplicate draft versions possible under concurrency

Two concurrent POST .../versions requests for the same active policy will both read sourcePolicy.version (e.g., 1) and both call createPolicy with version: 2. Because the version increment is purely in application code — not inside a database transaction with a uniqueness check — both writes succeed, leaving two draft records with the same version = 2 and the same supersedes pointer.

Later, when either draft is activated, activatePolicy will set the superseded policy to 'superseded' — but the sibling draft remains, creating an orphaned version entry with an invalid supersedes target.

To prevent this, either:

Add a unique composite index in SurrealDB on (workspace, supersedes) so a second concurrent CREATE fails immediately, or
Use a single atomic SELECT MAX(version) + 1 inside the CREATE statement so SurrealDB assigns the version number rather than the application layer.
