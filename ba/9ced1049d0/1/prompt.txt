Duplicate-draft guard misses testing status — two in-flight versions can be created

The duplicate-draft guard inside the transaction only blocks a second creation when a draft version already supersedes the same source policy:

LET $existing = (SELECT VALUE id FROM policy
    WHERE supersedes = $supersedes AND status = 'draft' LIMIT 1)[0];
IF $existing != NONE { THROW "a draft version already exists for this policy"; };
A policy in testing status is also an in-flight version that has not yet been activated. If a draft is created and later transitioned to testing via direct DB write or a future testing-promotion endpoint, the guard would pass and a second draft could be created superseding the same source policy. Both the testing version and the new draft would share the same supersedes pointer. When either is activated, activatePolicy sets $oldPolicy (the source) to superseded — leaving the other in-flight version pointing to an already-superseded policy.

Consider extending the guard to cover both statuses:

LET $existing = (SELECT VALUE id FROM policy
    WHERE supersedes = $supersedes AND status IN ['draft', 'testing'] LIMIT 1)[0];
IF $existing != NONE {
  THROW "an in-flight version already exists for this policy";
};
